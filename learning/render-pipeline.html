How vue-ink renders

…and why it does too much work. A visual walkthrough of the rendering pipeline, the wasted cells, and the fixes.

Fast UI frameworks have one trick: don't redo work for things that didn't change.
The whole question is where in the pipeline that idea is honored — and where it gets thrown away.

03The grid: what "rebuild the frame" actually means

vue-ink builds an in-memory grid of cells every paint, then stringifies it. Watch which cells light up on each tick.

What you'd want (browser-style)

Only the cells that actually changed are touched.

work per tick: ~1 cell

What vue-ink does today

Every cell is rewritten, stringified, and emitted.

work per tick: 80,000 cells

04The fix: per-row cache

Each row's output is a function of its cells. Hash the cells. If the hash matches last frame, reuse the cached string — don't restringify.

count: 5 a7f3 RECOMPUTED
row 0 8c21 CACHED
row 1 e019 CACHED
row 2 2bdc CACHED
row 3 5f7a CACHED
row 4 9d04 CACHED

05The deeper fix: skip the grid entirely

Most TUIs — chats, lists, forms — have no overlapping elements. So we don't need a 2D grid at all. We can just concatenate strings directly.

~2.6 ms

Grid path (today)

Walk every node
Write to W × H cell grid
Stringify every row
~0.3 ms

Flow path (new)

Render each subtree to string
Join with newlines

06The asymmetry

Look at the highlighted row. That's the entire bug. Vue earned the right to be lazy — we throw that work away in the paint step.

Take-away: Every improvement we can ship is a flavor of the same idea — teach the paint step to respect the dirty signal Vue already paid to compute. Either by caching what didn't change (per-row hashes, root flags), or by avoiding the work entirely (flow path). Once you carry that idea in your head, every line of the renderer makes sense.
